1 What is Microsoft Fabric?

Microsoft Fabric is an end-to-end, SaaS analytics platform that unifies everything you need to move, store, transform, analyse, and act on data — in one product, on one foundation called OneLake. It brings together the experiences formerly split across Power BI, Azure Synapse, and Azure Data Factory.

SaaS, not infrastructure

No clusters or servers to provision. Compute is auto-managed; you buy a single capacity and every workload draws from it.

OneLake at the centre

One logical data lake per tenant. All workloads read and write the same open Delta Parquet tables — one copy of the data.
AnalogyFabric is to analytics what Microsoft 365 is to productivity: separate apps (Word, Excel, PowerPoint) unified into one suite. Here, Data Factory, Lakehouse, Warehouse, and Power BI are unified on OneLake.

2 Core concepts

TermMeaning
CapacityThe compute you buy, sized as F SKUs (F2–F2048). Shared by every workload in its workspaces; can be paused to stop billing.
WorkspaceA collaborative container for items, bound to one capacity (and optionally a domain).
ItemAny artifact you create: lakehouse, warehouse, notebook, pipeline, semantic model, report, eventhouse, and more.
OneLakeThe single, tenant-wide data lake built on ADLS Gen2, stored in open Delta/Parquet format.
ShortcutA pointer to data elsewhere (another workspace, ADLS, S3, GCS, Dataverse) — queried in place, never copied.
WorkloadAn experience: Data Engineering, Data Factory, Data Warehouse, Data Science, Real-Time Intelligence, Power BI, Databases.
Direct LakeA Power BI storage mode that reads Delta tables straight from OneLake — import speed, no refresh.
DomainAn organisational grouping of workspaces for governance and a data-mesh approach.

3 Getting started

  1. An admin enables Fabric for the tenant in the Admin portal (a one-time tenant setting).
  2. Start a free 60-day trial or assign an F-SKU capacity (in Azure or via reserved capacity).
  3. Go to app.fabric.microsoft.com, create a workspace, and assign it to your trial or capacity.
  4. Pick a workload from the switcher (bottom-left) and create your first item — a Lakehouse is a great start.
TipThe free trial includes a trial capacity (~F64 equivalent) and 1 TB of OneLake storage, with no credit card required.
Watch your capacityAll workloads share the same compute pool. Heavy Spark jobs, big SQL queries, and refreshes compete for it — monitor with the Microsoft Fabric Capacity Metrics app to spot throttling.

5 OneLake — the data lake

There is exactly one OneLake per tenant, provisioned automatically — think "OneDrive for data". Every workspace is a container inside it, and every lakehouse or warehouse stores its tables there as open Delta Parquet.

  • One copy, many engines: Spark, T-SQL, KQL, and Power BI Direct Lake all read the same Delta tables — no duplication.
  • Shortcuts: virtualise data from ADLS, Amazon S3, Google Cloud Storage, Dataverse, or another workspace without moving a byte.
  • OneLake file explorer: a Windows app that syncs OneLake to your PC just like OneDrive.

OneLake paths follow a predictable, ABFS-style format:

abfss://<workspace>@onelake.dfs.fabric.microsoft.com/<item>.Lakehouse/Tables/<table>
abfss://<workspace>@onelake.dfs.fabric.microsoft.com/<item>.Lakehouse/Files/<path>
Shortcuts ≠ copiesA shortcut leaves data at its source and governs it once. Change the source and every Fabric engine sees the update — no pipeline, no sync lag.

6 Lakehouse

A lakehouse blends a lake's flexibility (any file, any format) with a warehouse's structure (Delta tables, SQL). It has two zones — Tables (managed Delta) and Files (raw/unstructured) — and automatically gives you a read-only SQL analytics endpoint plus a default semantic model.

  1. Get data in: upload, create a shortcut, or run a pipeline/dataflow into Files.
  2. Transform it in a notebook (Spark) or dataflow.
  3. Save the result as a managed Delta table.
  4. Query it with the SQL endpoint, or build a Power BI report on it.
# PySpark in a Fabric notebook with a lakehouse attached
df = spark.read.option("header", True).csv("Files/raw/sales.csv")
clean = df.dropna().withColumnRenamed("amt", "amount")
clean.write.format("delta").mode("overwrite").saveAsTable("sales_clean")
Medallion architectureOrganise data in layers — Bronze (raw) → Silver (cleaned/conformed) → Gold (business-ready) — using schemas or separate lakehouses.

7 Data Warehouse

The Fabric Warehouse is a full, ACID-compliant, read/write T-SQL warehouse — but its data lives in OneLake as open Delta, so it's cross-queryable by every other engine. Choose it when you want a SQL-first experience and need to write with T-SQL (the lakehouse SQL endpoint is read-only).

LakehouseWarehouse
Primary engineSpark (+ read-only T-SQL endpoint)T-SQL (read & write)
Best forData engineers, data science, filesSQL / DW developers, BI marts
Load withNotebooks, pipelines, dataflowsCOPY INTO, T-SQL, pipelines
-- T-SQL in a Fabric Warehouse
CREATE TABLE dbo.sales (id INT, amount DECIMAL(10,2), region VARCHAR(40));

COPY INTO dbo.sales
FROM 'https://<account>.blob.core.windows.net/data/sales.parquet'
WITH (FILE_TYPE = 'PARQUET');

-- Cross-query a lakehouse table from the warehouse, in one statement
SELECT w.region, SUM(w.amount) AS total
FROM dbo.sales AS w
JOIN [MyLakehouse].dbo.regions AS r ON r.code = w.region
GROUP BY w.region;

8 Data Factory

Fabric's Data Factory moves and transforms data. Four tools cover most needs:

Pipelines

Orchestration & control flow with 200+ connectors. The Copy activity does scaled-out data movement.

Dataflows Gen2

Low-code, Power Query transforms — the same experience as Power BI dataflows, landing in OneLake.

Copy job

A simplified, guided way to move data (batch or incremental) without building a full pipeline.

Mirroring

Near-real-time replicas of Azure SQL, Snowflake, Cosmos DB, and more into OneLake — free, no ETL.
  1. Create a Data pipeline item.
  2. Add a Copy data activity; pick a source connector and a OneLake destination.
  3. Add transforms (notebook, dataflow, stored proc) downstream.
  4. Attach a schedule or event trigger.
Mirroring firstTo get operational data into OneLake continuously, try Mirroring before building ETL — it keeps a live Delta copy you can query with any Fabric engine.

9 Notebooks & Spark

Fabric notebooks run on managed Spark and support PySpark, Spark SQL, Scala, and R. Attach a lakehouse, and a starter pool spins up a session in seconds. Environments let you pin libraries and Spark configuration for reuse.

# Mix languages with cell magics
df = spark.read.format("delta").load("Tables/sales_clean")
df.createOrReplaceTempView("sales")
%%sql
SELECT region, SUM(amount) AS total
FROM sales GROUP BY region ORDER BY total DESC
# notebookutils: handy helpers for files, secrets, and orchestration
import notebookutils
files = notebookutils.fs.ls("Files/raw")
notebookutils.notebook.run("Transform_Silver", timeout_seconds=600)
Fast startsStarter pools give sub-10-second session start for quick work. Use custom pools and Environments when you need specific libraries or tuned Spark settings.

10 Power BI & Direct Lake

Power BI is a native Fabric workload. A semantic model defines your measures and relationships; reports sit on top. There are three storage modes — and Fabric adds a powerful third one.

ModeHow it worksTrade-off
ImportData cached in-memory in the model.Fastest queries, but needs scheduled refresh.
DirectQueryEvery visual queries the source live.Always fresh, but slower and source-dependent.
Direct LakeReads Delta tables straight from OneLake.Import-like speed and live freshness, no refresh.
Direct Lake fallbackIf a query exceeds capacity guardrails, Direct Lake transparently falls back to DirectQuery so reports keep working — you just lose a little speed for that query.

11 Real-Time Intelligence

For streaming, event, and time-series data, Real-Time Intelligence (RTI) provides ingest-to-insight in one workload. You query it with KQL (Kusto Query Language).

Eventstream

No-code ingest, transform, and route of events from Event Hubs, IoT, Kafka, and more.

Eventhouse

A database optimised for real-time analytics; it contains one or more KQL databases.

Real-Time Dashboard

Live, auto-refreshing tiles built on KQL queries.

Activator

Trigger actions when conditions in the stream are met (see §12).
// KQL: 5-minute event volume over the last hour, as a timechart
Events
| where Timestamp > ago(1h)
| summarize count() by bin(Timestamp, 5m)
| render timechart
Best of both worldsEventhouse auto-indexes for high-volume, low-latency time-series — and the data is also exposed in OneLake as Delta, so the rest of Fabric can use it too.

12 Data Activator

Data Activator (Activator) is a no-code engine that watches data and acts on it. It monitors Power BI visuals, Eventstreams, and KQL queries, then fires an action — email, Teams message, Power Automate flow, or a Fabric pipeline — when a condition is met.

  1. Pick a source (a Power BI measure, an Eventstream, or a KQL query).
  2. Define a condition — e.g. "temperature > 80° for 5 minutes" or "the query returns ≥ 1 row".
  3. Choose an action and recipients.
Pattern: alert on any matchA common trigger is "fire when a KQL query returns at least one row" — ideal for anomaly, threshold, and SLA-breach alerts without writing alerting code.

13 Governance & security

Security in Fabric is layered, from the capacity down to a folder in OneLake.

LayerWhat it controls
Workspace rolesWho can admin, edit, or view items (see §4).
Item permissionsSharing a single report, lakehouse, or model independently.
OneLake data-access rolesRBAC on folders/tables within a lakehouse.
Sensitivity labels (MIP)Classification that flows with items and into exports (PDF, Excel, PBIX).

Govern the sourceLabels and permissions on a shortcut don't automatically protect the underlying external data. Secure the source system as well, and use workspace identities / managed private endpoints for private connectivity.

14 Git integration & CI/CD

Fabric workspaces connect to Azure DevOps or GitHub. Items serialise to source-control-friendly definitions, so you can branch, review, and promote — exactly like the workflow in the GitHub manual.

  1. Open Workspace settings → Git integration and connect a repo and branch.
  2. Commit your items to the branch; pull updates to sync incoming changes.
  3. Use a feature branch + pull request for review, then merge.
  4. Promote across stages with a deployment pipeline (Dev → Test → Prod).
git switch -c feature/new-semantic-model   # branch your changes
# ...commit item definitions in Fabric, they appear in the repo...
git push -u origin feature/new-semantic-model
gh pr create --fill                        # open a PR for review
Git + pipelines togetherUse Git for source control and reviews, and deployment pipelines for stage promotion. Parameterise connections with deployment rules so Prod points at Prod sources.

15 Quick cheat sheet

KQL essentials (Real-Time Intelligence)

T | take 100                      // peek at rows
T | where Ts > ago(1d)            // filter by time
T | summarize count() by Region   // aggregate
T | summarize avg(v) by bin(Ts,1h)// time buckets
T | make-series c=count() default=0 on Ts step 1h
T | join kind=inner U on Key      // join two tables
  | render timechart               // visualise

Lakehouse / PySpark

df = spark.read.format("delta").load("Tables/t")
df.write.format("delta").mode("append").saveAsTable("t")
spark.sql("MERGE INTO t USING s ON t.id=s.id "
          "WHEN MATCHED THEN UPDATE SET * "
          "WHEN NOT MATCHED THEN INSERT *")

Warehouse / T-SQL

COPY INTO dbo.t FROM '<url>' WITH (FILE_TYPE='PARQUET');
CREATE TABLE dbo.g AS SELECT region, SUM(amt) a FROM dbo.t GROUP BY region;
