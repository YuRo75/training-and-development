1 What is GitHub?

Git is a version-control system that tracks changes to files over time. GitHub is a cloud platform built on top of Git that hosts your repositories and adds collaboration tools: pull requests, issues, code review, automation, and project management.

Git (the tool)

Runs locally on your machine. Tracks history, lets you branch and merge, works offline.

GitHub (the platform)

Hosts repos in the cloud, enables teams to collaborate, review code, and automate workflows.
AnalogyGit is like the "track changes" engine; GitHub is the shared office where everyone's documents live and where you discuss and approve edits.

2 Core concepts

TermMeaning
Repository (repo)A project's folder, including all files and their complete history.
CommitA saved snapshot of changes with a message describing what changed.
BranchAn independent line of work, so you can develop without affecting main.
MergeCombining the changes from one branch into another.
CloneDownloading a full copy of a repo to your machine.
ForkYour own server-side copy of someone else's repo.
Pull request (PR)A proposal to merge your branch, with review and discussion.
IssueA tracked task, bug report, or feature request.
RemoteA hosted version of your repo (e.g., on GitHub), usually named origin.

3 Account & setup

  1. Create a free account at github.com.
  2. Install Git from git-scm.com (or use winget install Git.Git on Windows).
  3. Tell Git who you are: 
    git config --global user.name "Your Name"
git config --global user.email "you@example.com"
  4. Authenticate. The easiest path is the GitHub CLI: 
    winget install GitHub.cli
gh auth login   # follow the browser prompts
TipPrefer SSH or the GitHub CLI over typing your password. Personal access tokens replaced password auth for Git operations years ago.

4 Your first repository

Option A — start on GitHub

  1. Click New on your repos page, name it, and add a README.
  2. Clone it locally: 
    git clone https://github.com/<you>/<repo>.git
cd <repo>

Option B — start locally, push up

mkdir my-project && cd my-project
git init
echo "# My Project" > README.md
git add .
git commit -m "Initial commit"
gh repo create my-project --private --source=. --push

5 Git basics: the daily loop

Almost every day with Git is the same short cycle:

git status              # see what changed
git add file.txt        # stage a file (or: git add .)
git commit -m "message" # save a snapshot
git pull                # get teammates' latest changes
git push                # upload your commits
The three areasFiles move from your working directory → staging area (git add) → repository (git commit) → remote (git push).

Writing good commit messages

  • Use the imperative mood: "Add login validation", not "Added" or "Adds".
  • Keep the summary under ~50 characters; add detail in the body if needed.
  • One logical change per commit.

6 Branching

Never build directly on main. Create a branch per feature or fix:

git switch -c feature/login   # create + switch to a new branch
# ...make changes, add, commit...
git push -u origin feature/login
ActionCommand
List branchesgit branch
Create & switchgit switch -c name
Switch existinggit switch name
Delete merged branchgit branch -d name
Naming conventionUse prefixes like feature/, fix/, docs/, chore/ so branches are self-describing.

7 Pull requests (PRs)

A PR is how your branch gets reviewed and merged. The flow:

  1. Push your branch to GitHub.
  2. Open a PR (button on the repo page, or gh pr create).
  3. Describe what and why; link any related issues with Closes #123.
  4. Reviewers comment and approve; CI checks run automatically.
  5. Merge (squash, merge, or rebase), then delete the branch.
gh pr create --fill            # open a PR from current branch
gh pr status                   # see your PRs
gh pr checks                   # view CI status
gh pr merge --squash --delete-branch
Review etiquetteKeep PRs small and focused. A 200-line PR gets a real review; a 2,000-line PR gets a rubber stamp.

8 Forking (contributing to others' projects)

Use a fork when you don't have write access to a repo, e.g., open-source contributions.

  1. Fork the repo (button on GitHub, or gh repo fork owner/repo --clone).
  2. Add the original as an upstream remote to stay in sync: 
    git remote add upstream https://github.com/owner/repo.git
git fetch upstream
git merge upstream/main
  3. Create a branch, commit, push to your fork.
  4. Open a PR from your fork's branch back to the original repo.
Fork vs. branchA branch lives inside one repo (use it when you have write access). A fork is a separate copy under your account (use it when you don't).

9 Collaboration tools

Issues

Track bugs, tasks, and ideas. Use labels, assignees, and milestones to organize.

Projects

Kanban-style boards that link issues and PRs to plan and track work.

Code review

Inline comments on diffs, suggested changes, and required approvals before merge.

Discussions

Q&A and open-ended conversation separate from actionable issues.

Protecting main

Repo admins add branch protection rules: require PR reviews, passing CI checks, and no direct pushes to main. This keeps the default branch always shippable.

10 GitHub Actions (automation & CI/CD)

Actions run automated workflows on events like a push or PR. Define them in .github/workflows/*.yml.

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm test
Common usesRun tests, lint code, build artifacts, deploy on merge, auto-label PRs, or publish releases.

12 Quick cheat sheet

# Setup
git clone <url>                 clone a repo
git remote -v                   list remotes

# Daily loop
git status                      what changed
git add .                       stage everything
git commit -m "msg"             save snapshot
git pull                        fetch + merge remote
git push                        upload commits

# Branching
git switch -c name              new branch
git switch main                 change branch
git merge name                  merge into current
git branch -d name              delete branch

# Undo
git restore file                discard local changes
git restore --staged file       unstage
git reset --soft HEAD~1         undo last commit, keep changes
git revert <hash>               new commit that undoes a commit
